Part 2: Applied Theory

Goals for our (S)CSS

  • Well-organized
  • Reusable
  • Maintainable
  • Portable

Implementing a CSS methodology will almost certainly require an adjustment to how you write markup.

OOCSS - Object-Oriented CSS

One of the first attempts at standardizing an approach to reusable objects

Achieves reusability by:

  • Separate structure from skin
  • Separate content from container
See an example

OOCSS

Pros:

  • Very flexible
  • Non-prescriptive code approach

Cons:

  • Very few guide rails for beginners
  • Has become a bit passé against new approaches

SMACSS - Scalable and Modular Architecture for CSS

Advocates a breakdown of concerns for CSS

  • Base Rules
    • Global rules, normalize, resets
  • Layout Rules
    • High-level structural styles
  • Module Rules
    • Styles for specific modules
  • State Rules
    • Describes state of modules: hover, clicked, selected
  • Theme Rules
    • Design-specific styles, usually if multiple designs are available

SMACCS

Pros:

  • Structured and organized
  • Reusability
  • Pseudo-name-spacing solution
    • .module
    • .module-element

Cons:

  • Can get overly complex
  • Related styles can get separated
  • Doesn’t take name-spacing far enough to solve scope

BEM - Block--Element__Modifier

Name-spacing solution

  • .human
  • .human__arm
  • .human__arm--left
  • .human--female
  • .human-female__arm
  • .human-arm__left-finger

BEM

Pros:

  • Very contextual
  • Keeps the specificity very low
  • Enterprise-project friendly
  • Large dev team friendly
  • Class names provide context between markup and style

Cons:

  • Can feel like a large investment on small projects
  • Complex selectors can feel overwhelming
  • Portability requires forethought
  • Long class names make developers feel icky
  • Combining underscores and hyphens makes developers feel icky
